# OpenBYD 2.2 → Yandex patch

**Language:** [Belarusian](../openbyd-patch/PATCH_NOTES.md) · **English** · [Russian](PATCH_NOTES.ru.md)

## What OpenBYD 2.2 actually does (from decompile)

- It is **not** a receiver of `AUTONAVI_STANDARD_BROADCAST_SEND` — it *sends* that broadcast
  (see `HudController.sendStandardAmapBroadcast`). The original “bridge” plan (Yandex → my app →
  AUTONAVI broadcast → OpenBYD) cannot work, because OpenBYD never listens for that broadcast.
- Its **only** navigation input is `MapNotificationListenerService`, a NotificationListenerService
  whitelisted to `com.google.android.apps.maps` and `app.revanced.android.apps.maps`.
- It drives the BYD cluster via a privileged path: `ProxyManager` → `ICarControl.turnOnNavi()` /
  `updateNavigation()` + `ClusterProjectionService`. That privileged path is the reason we patch
  OpenBYD rather than reimplement it.

### Parser facts (MapNotificationListenerService.onNotificationPosted)
- Reads `android.title`, `android.text`, `android.subText`; requires `flags & FLAG_ONGOING_EVENT (2)`.
- Distance-to-maneuver regex on **title**: `(\d+[\.,]?\d*)\s*(m|м|km|км)` — already Cyrillic-aware.
- ETA/route-remaining regex on **subText** already includes `ч|мин|км|м` — already Cyrillic-aware.
- **Maneuver direction** comes ONLY from the notification large-icon bitmap, matched against a
  perceptual-hash registry of Google Maps' arrow icons (`w40.c`), with a geometric left/right/straight
  fallback. Yandex's icons will NOT match the registry → only the coarse geometric fallback applies.
- Builds `g60(iconId:int, distance:Integer, road:String, routeDist:Integer, routeTime:Integer)`
  → `HudController.updateNavigation(context, g60, transliterate:boolean)`.

## Phase 1 patch (applied) — whitelist Yandex

File: `smali/com/sr/openbyd/services/MapNotificationListenerService.smali`, `<clinit>`:
- `.locals 2` → `.locals 3`
- added `const-string v2, "ru.yandex.yandexnavi"`
- `filled-new-array {v0, v1}` → `filled-new-array {v0, v1, v2}`

This makes OpenBYD accept Yandex notifications and run its existing (Cyrillic-aware) distance/ETA
parser + the geometric icon fallback. Expect: distance/road/ETA likely OK; turns coarse (L/R/straight).

## Phase 1b patch (applied) — file logger (no adb needed)

Added `smali/com/sr/openbyd/YBLog.smali` (source kept here as `YBLog.smali`), a static logger that
appends one line per captured notification to:

    /sdcard/Android/data/com.sr.openbyd/files/yandex_nav_log.txt

(written via `Context.getExternalFilesDir(null)` — no runtime permission required). Readable on the
head unit with any file manager.

Injected one call in `MapNotificationListenerService.onNotificationPosted`, at the `:goto_0` label
right after title/text/subText are resolved (registers v1=this(Context), v6=title, v9=text, v8=subText):

    invoke-static {v1, v6, v9, v8}, Lcom/sr/openbyd/YBLog;->w(Landroid/content/Context;Ljava/lang/String;Ljava/lang/String;Ljava/lang/String;)V

Each line: `<Date> | T=<title> | X=<text> | S=<subText>`. Logs every whitelisted nav notification
(before the dedup/filter steps), so duplicates are expected — that's fine for format capture.

## Phase 2 (pending) — precise maneuver mapping

Needs Yandex's real notification field layout (see CAPTURE below). If the maneuver fidelity from the
geometric fallback is insufficient, inject a Yandex-specific keyword→iconId mapper (RU/EN/TR) that
overrides the icon path for the Yandex package. iconId values are the `TURN_ICON_*` ints in
`HudController` (1=LEFT, 2=RIGHT, 3/4=slight L, 5/6=slight R, 7=sharp L, 8=sharp R, 9/10=U-turn,
11/12=straight, 15-20=roundabout, 48=destination).

## Workspace layout (lean — decoded/ is generated, not committed)

    openbyd-patch/
      original/OpenBYD-2.2-orig.apk   pristine input (git-ignored; keep your own copy)
      patches/YBLog.smali             source of the injected file-logger class  [tracked]
      apply_patches.py                applies the edits below to a decoded tree  [tracked]
      build.sh                        decode -> patch -> build -> sign -> verify [tracked]
      PATCH_NOTES.md                                                             [tracked]
      decoded/                        apktool output — GENERATED, git-ignored
      dist/OpenBYD-2.2-yandex.apk     build output — git-ignored
      yandexbyd.keystore              throwaway signing key — git-ignored

Only `original/`, `patches/`, the two scripts, and this file are tracked. Everything else is
regenerated by `build.sh`, so the repo stays small and the patch logic is the source of truth.

## Build recipe

    ./build.sh            # decode pristine + apply patches + sign  (reproducible)
    ./build.sh --keep     # skip re-decode, re-patch existing decoded/ + sign (faster iteration)

Tools: apktool, Android build-tools (zipalign, apksigner), JDK keytool, python3. The keystore is
auto-created on first run (alias `yb`, storepass/keypass `android`).

`apply_patches.py` is idempotent and asserts each edit matches exactly once, so a new apktool version
or a new OpenBYD build fails loudly (anchor mismatch) instead of producing a broken APK. The edits it
applies are exactly the four described above (clinit .locals, whitelist string, 3-elem array, logger
call) plus copying `patches/YBLog.smali` into place.

To hand-edit instead: run `./build.sh` once (to populate `decoded/`), edit smali there, then
`./build.sh --keep`.

NOTE: re-signing changes the APK signature, so the original OpenBYD must be uninstalled before
installing this build (you cannot update-in-place across different signing keys).
